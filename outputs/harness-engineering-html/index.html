Ralph Loop09
Stateless Agent Loop

Ralph Loop大道至简的 harness 尝试

状态外置于文件系统，Agent 只负责一次性推进

工作机制

脚本只负责重启；世界负责记忆。

一句话定义 让 AI 编码 Agent 在 shell 死循环里反复执行同一个 prompt；每轮都用全新上下文，状态完全落在代码库、TODO 文件、git history 上。

读 prompt.md同一个任务入口
改代码 / 写测试Agent 推进一轮
提交每轮一个 commit
退出上下文释放
检查 COMPLETE未完成则全新 context 再来
核心意义 · 无状态化

从单机 session，变成 stateless 服务。

有状态 Agent 单机 session context 越长越脆，崩了就全没。
→
无状态 Agent 外部存储 (prompt, 磁盘状态) → 新的磁盘状态
“记忆”不再是 LLM 的负担，而是工程问题：交给 git、文件、AGENTS.md。
为什么必须无状态
  • LLM 的 “malloc 没有 free”：context 只进不出，长会话注定劣化。
  • 无状态 = 可重试、可并行、可中断、可审计。
  • 前提：可靠反馈回路 + 明确完成信号。
不要让 Agent 记住任务，让任务自己留在世界里。
Stateless Agent Loopprompt.md · git · TODO · CI · COMPLETE

收束23
本场收束

驾驭工程本质上是基于大模型的软件工程。

真正稀缺的不再是多写几句提示词，而是定义任务边界、设计反馈回路、控制权限和风险、把失败沉淀成基础设施。

工程师不会消失。

工程师的重心会上移：建设让智能体可以可靠工作的基础设施。
