本文件根據李宏毅教授《AI Agent (1/3)：核心技術 Context Engineering 基本概念解說》視頻內容整理而成。

# Part 01: 引言與 AI Agent 的基本定義

各位同學大家好
今天我們要進入 AI Agent 系列課程的第一部分
這堂課的內容會比較偏向科普性質
但是會引用非常多最新的論文
內容主要分為三個大塊：
1. **核心技術：Context Engineering**
2. **AI Agent 之間的互動方式**
3. **AI Agent 對未來工作可能帶來的衝擊**

首先我們來定義一下什麼是 AI Agent
大家過去在使用 AI 的時候
往往是一個指令一個動作
這叫做「傳統 AI 的使用方式」
而 AI Agent 翻譯過來就是「人工智慧代理人」
它與傳統 AI 的最大差別在於
人類給予的是一個「目標 (Goal)」
而 AI 會自己想辦法達成這個目標
這中間通常會包含一個循環
我們稱為 **Goal -> Action -> Observation** 的循環
首先由人類給定一個 Goal
Agent 會去觀察 (Observation) 目前的環境狀況
根據觀察結果採取行動 (Action)
行動會改變環境，產生新的 Observation
Agent 再根據新的狀況決定下一個 Action
直到最終達成目標為止

舉例來說，如果你叫 AI 幫你訂一張去日本的機票
傳統方式你得告訴它去哪個網站、選哪一班、填什麼資料
而 Agent 的方式是，你只給它目標
它會自己去 Google 搜尋、比價、判斷剩餘張數、甚至處理報錯
這就是 Agent 的魅力所在
而在這堂課中，我們不討論如何重新訓練模型
我們要討論的是如何透過 **Context Engineering (上下文工程)**
讓現有的強大語言模型 (LLM) 就能具備這樣的能力
我們會發現，AI 的表現上限往往不在於模型本身的權重
而是在於我們如何設計輸入給它的資訊，也就是「上下文」

# Part 02: 從 Prompt 到 Context Engineering

我們過去常聽到一個詞叫做 Prompt Engineering
也就是提示工程，很多人戲稱為「咒語」
在語言模型能力還沒那麼強的時候
你加一句 "Let's think step by step" 模型就突然變聰明了
但如果你有在關注最新的模型，你會發現這些咒語的效果正在減弱
因為今天的模型，你不叫它認真思考，它也會發揮全力
那「咒語」失效之後，我們接下來要面對的是什麼？
那就是 **Context Engineering (上下文工程)**
它不再只是改一句話，而是管理整個輸入給模型的「脈絡」

Context Engineering 更像是一張「施工藍圖」
它決定了在每一輪的對話中，模型應該看到什麼
我們可以把 Context 拆解成幾個組成要素：
1. **System Prompt**：規範角色的性格、底線，這部分現在越來越長
2. **User Input**：使用者的具體需求
3. **Dialogue History**：短期的對話紀錄
4. **Memory**：從長期記憶庫中檢索出來的相關資訊
5. **RAG (External Info)**：外部工具搜尋到的資料
6. **Tool Output**：工具執行的結果
7. **Reasoning**：模型內部的推理過程

這就是為什麼我們說 Context Engineering 是更有系統的設計
它要解決的是模型「活在當下」的問題
語言模型本質上是無狀態的 (Stateless)
它之所以能進行多輪對話，是因為 we 在背後偷偷把之前的對話都傳回去了
而如何優雅地管理這些不斷增長的資訊，就是這堂課的核心

# Part 03: 記憶管理的核心模組

當 AI Agent 運作一段時間後，它產生的資訊會非常驚人
如果你把所有的 Observation 和 Action 都丟進去
你的 Context Window 很快就會被塞爆
所以我們需要一套記憶管理系統
李宏毅老師將其拆解為三個核心模組：

1. **Read (讀取模組)**：
   這很像我們現在常說的 RAG (檢索增強生成)
   Agent 不會把所有歷史紀錄都丟進去
   而是根據目前的狀況，從「長期記憶資料庫」中篩選出最相關的片段
   
2. **Write (寫入模組)**：
   這是一個「判斷」的過程
   AI 會問自己：「這件事值得記下來嗎？」
   它會過濾掉瑣碎的雜訊，只將重要的資訊存入長期記憶
   
3. **Reflection (反思模組)**：
   Agent 會定期對過去的經驗進行「總結」或「反思」
   它會把零碎的 Observation 轉化為高層次的的想法或計畫
   這不僅節省了空間，更提升了模型的決策品質

透過這三個模組，我們就能解決模型「資訊過載」的問題
讓它在有限的 Context 內，依然能根據精挑細選過的記憶來做出精準的決策
這就是為什麼一個好的 Agent 系統，其背後的 Context 設計是非常複雜的

---

# Part 04: 工具呼叫與 Function Calling

AI Agent 另一個強大的地方在於它會使用工具
這在技術上我們稱為 **Function Calling**
對 LLM 來說，使用工具其實就是輸出一段特定格式的文字
比如它輸出一段 JSON，告訴系統它想用 `GoogleSearch` 這個 Function
外部系統執行完後，再把結果 (Observation) 餵回給模型

在這裡，李老師提到了一個非常有趣的觀察：
模型其實是有自己的「判斷力」的
當外部工具提供的資訊與模型內部的知識發生衝突時，模型並不會盲目相信工具
它會去比較、去推理
例如，如果搜尋結果說今天是 2026 年，而模型內部知識停在 2024 年
模型通常會意識到「外部資訊是更新的」，進而採納工具的建議
但如果工具回傳的資訊過於荒謬，模型也可能會選擇不相信

這告訴我們，Context Engineering 的另一個任務
就是提供足夠的「證據」和「邏輯鏈」
讓模型能在衝突的資訊中做出正確的判斷
這不再只是簡單的文字拼接
而是在設計一套資訊互動的規範
讓模型能在多變的環境下依然保持理智

---

# Part 05: 上下文視窗與「超憶症」比喻

接下來我們來聊聊 Context Window 的限制
雖然現在有些模型號稱有 100 萬、200 萬個 Token 的容量
但這不代表你可以無限制地塞東西進去
因為這會帶來兩個嚴重的問題：
1. **算力成本與速度**：Token 越多，運算越慢，錢也燒得越快
2. **Lost in the Middle**：研究發現，當資訊放在中間時，模型很容易忽略它

這裡李老師用了一個非常有意思的醫學比喻：**「超憶症 (Hyperthymesia)」**
超憶症患者能記住生命中每一天的所有細節
你可能會覺得這很棒，但實際上這是一種負擔
因為這些人往往難以進行抽象思考
他們的大腦被過多無意義的雜訊（比如十年前的早餐吃了什麼）塞滿了
導致在做決策時，無法從浩如煙海的細節中抽離出來

對 AI Agent 也是一樣
如果你讓它記住每一個瑣碎的步驟，它會變得很笨
它會陷入冗長的回憶中，而無法針對當前情境做出判斷
這就是為什麼我們需要 Context Engineering
去人為地製造「遺忘」和「篩選」
這不是在削減模型的能力，而是在優化它的智力
這也解釋了為什麼「資訊管理」在 Agent 時代變得比「訓練模型」更重要

---

# Part 06: 主動壓縮與 Subagent 隔離

既然 Context 太長會有問題，那我們該如何壓縮它？
傳統的做法是開發者寫死一些規則（比如每 10 輪就摘要一次）
但李老師提出了一個更聰明的觀點：**「主動壓縮」**
我們能不能讓模型自己決定什麼時候該壓縮資訊？

這裡有一個非常巧妙的手段：**「呼叫 Subagent (子代理)」**
大家可能會覺得呼叫 Subagent 是為了分工
但在 Context Engineering 的視角下，它其實是一種**高效的壓縮機制**
為什麼呢？
當 Main Agent 呼叫 Subagent 去做一件「髒活」時
Subagent 在執行過程中可能會產生幾萬個 Token 的日誌和對話
但當 Subagent 做完之後，它只會回報一個簡短的結論給 Main Agent
對於 Main Agent 來說，那些繁瑣的執行細節都被「隱藏」掉了
它只拿到了精簡後的資訊
在效果上，這就達成了一種非常精準的 Context 壓縮
同時也實現了「資訊隔離」
主模型的 Context 始終保持乾淨，能把注意力集中在最高層次的規劃上
這就是為什麼現在越來越多的系統採用 Multi-Agent 架構
這不僅是工程上的分工，更是對 Context 的極致管理

---

# Part 07: Context Collapse 與 ACON 論文

然而，壓縮資訊是有代價的
最嚴重的問題就是 **Context Collapse (上下文崩潰)**
這指的是在摘要或壓縮的過程中，模型不小心遺失了關鍵資訊
舉例來說，System Prompt 裡規定「必須經過人類同意才能刪除檔案」
但在摘要過後，這條關鍵指令被模型當成「細節」給去掉了
導致模型在接下來的行動中開始擅作主張

為了解決這個問題，李老師介紹了一篇最新的論文：**ACON (Agentic Context Optimization)**
ACON 的核心思想是利用「反饋機制」來優化摘要
它會讓另一個模型去檢查生成的摘要
看看這個摘要是否足以支持接下來的任務
如果摘要不夠好，它會分析原因並回頭修正摘要策略
這讓摘要不再是單向的「丟棄資訊」
而是一個雙向的「確保關鍵資訊留存」的動態過程

這再次證明了 Context Engineering 正朝著自動化和系統化發展
我們不再是憑感覺寫 Prompt
而是建立一套闭環系統，讓 Context 在傳遞過程中能自動保持完整與精煉
這對於開發長對話或複雜任務的 Agent 系統至關重要

---

# Part 08: 強化學習優化摘要 (SUPO)

除了 ACON 之外，還有一篇非常厲害的論文叫做 **SUPO (Summarization augmented Policy Optimization)**
SUPO 解決的是一個更難的問題：當我們不知道「正確的摘要」長什麼樣子時，該怎麼辦？
在很多複雜任務中，我們很難定義哪些資訊是絕對重要的
SUPO 的做法是利用**強化學習 (Reinforcement Learning)**

具體的流程是這樣的：
1. 讓模型嘗試產生不同版本的摘要
2. 讓 Agent 根據這些摘要去解題
3. 如果 Agent 最後把題目解對了，就給該摘要正向獎勵
4. 如果解錯了，就給負向獎勵

透過這種方式，模型會慢慢學會：
「原來在這種情境下，保留這幾行程式碼比保留那一串對話更重要」
這不僅提升了模型的摘要能力
更讓 Agent 學會了如何在資訊有限的情況下執行任務
這就是 Context Engineering 的最高境界
讓模型具備「洞察重點」的能力，而這種能力是可以透過強化學習不斷進化的
這也讓 AI Agent 從一個「聽命行事」的工具，變成了一個能「自我優化」的智慧體

---

# Part 09: 觀察遮蓋 (Observation Masking)

當 AI Agent 在進行軟體工程或處理大量數據時
它會呼叫工具並得到非常長的輸出，我們稱為 **Observation (觀察)**
研究顯示，在沒有優化的情況下，這些 Observation 佔據了約 84% 的 Token 消耗
特別是在 Read Code (讀取程式碼) 的任務中
有些程式碼幾千行，一次 Read 就會塞滿 Context

為了解決這個問題，工程師們發展出了 **Observation Masking (觀察遮蓋)** 技術
它的核心邏輯是：
1. 模型不需要每次都看到完整的工具輸出
2. 我們可以用簡單的佔位符 (Placeholder) 或總結來取代長輸出
3. 只有當模型明確要求「我想看某一段細節」時，才把該內容抓進 Context

這就像是給模型看一份「目錄」而不是整本書
模型可以先瀏覽目錄，決定去讀哪一章
這極大地節省了 Token 成本，也避免了模型被大量無關的程式碼干擾
分析發現，透過這種精細的遮蓋與篩選
我們可以用極少的 Token 就讓模型完成原本需要巨大 Context 才能做到的事
這也是 Context Engineering 中最實用的「省錢」招數

---

# Part 10: Multi-Agent 與 Context 隔離

隨著任務複雜度的提升，單一 Agent 的負擔會越來越重
它要同時記住高層次的 Goal
又要處理底層的 Tool Log
還要應付各種報錯
這會導致 Context 變得非常混亂，甚至出現「精神分裂」
為了解決這個問題，李老師提出了一個核心策略：**Context 隔離**

具體做法是將大 Agent 拆分成多個小 Agent：
*   **Planner (總招)**：只負責規劃和任務分配，它的 Context 非常乾淨，只包含大目標和進度
*   **Executor (執行者)**：負責具體操作，它的 Context 包含詳細的 Tool Output
*   **Evaluator (評估者)**：負責檢查結果，它的 Context 包含驗證標準

透過這種「分工」，每個 Agent 只需要處理自己那一小塊 Context
資訊在傳遞時經過了「人為的過濾與摘要」
主模型不再需要看到所有的髒活細節
這種「隔離」不僅解決了長度限制問題
更提升了系統的穩定性
這也解釋了為什麼現在的 AI 應用越來越傾向於用一群模型
而不是一個超級模型來解決所有問題
管理這群模型的資訊流，就是 Context Engineering 的高級課題

# Part 11: 評量的挑戰與 AI Judge 偏見

在開發完 Context Engineering 系統後，我們該如何評估它？
這裡李老師介紹了一個有趣的 Benchmark 叫做 **ToolBench**
在 ToolBench 中，AI Agent 要扮演客服去處理人類的需求（比如訂票、退貨）
但有趣的是，為了自動化測試，這裡的「人類」其實也是另一個 AI 假扮的

這就帶來了一個嚴重的問題：**評估偏差**
研究發現，AI 扮演的客戶往往「太客氣」且「描述太清晰」了
真實的人類說話可能很簡短、語氣不好、資訊缺漏
但 AI 客戶會把名字、訂單編號、需求講得一清二楚
這導致我們在測試中得到的成功率，往往會高估了現實中的性能

此外，我們通常還會用第三個模型來當 **AI Judge (評審)**
李老師指出，AI Judge 有一種潛在的偏見：
它傾向於給予「更像 AI 的表現」高分
如果你回答得溫文儒雅、條理分明（雖然可能沒解決問題）
AI Judge 就會覺得你很棒
這種「模型間的相互吹捧」是目前 AI Agent 評量的一大難點
這告訴我們，雖然自動化評量很快
但最終我們還是需要真實世界的數據和人類的直覺來做最後的把關

# Part 12: 總結與未來展望

總結今天的課程
我們學到了一個最重要的觀念：
**「在 AI Agent 時代，Context Engineering 就是新的 Prompt Engineering」**
我們不再軍事是尋找神奇的咒語
而是要像系統工程師一樣，有系統地管理模型的輸入資訊

同一個模型，只要你設計好 Read/Write/Reflection 模組
只要你懂得如何進行主動壓縮與觀察遮蓋
只要你能優雅地進行多代理人的 Context 隔離
它就能展現出原本不具備的強大智慧
模型是固定的（權重不變）
但 Context 是無限可能的（施工方式由你決定）

展望未來
李老師認為 AI Agent 將不再只是一個一次性的工具
它會成為陪伴人類一輩子的夥伴
它會具備長期演進的能力
透過不斷地在 Context 中積累經驗、優化技能
它會跟著主人一起成長
這條路還很長，Context Engineering 也才剛剛起步
希望大家能從今天的課程中
感受到這種「駕馭 AI」的新藝術
謝謝大家
